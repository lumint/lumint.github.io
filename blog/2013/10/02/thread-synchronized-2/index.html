互斥量:

      在Java中互斥量必须为对象类型,不能是基本数据类型,在Java中互斥量应该被称为锁.在一个应用中什么时候使用锁呢? 该如何使用锁呢?
      先看下面这个计数例子:

      上面程序因为没有使用同步机制,所以运行结果一般会小于20000,这是因为在increment方法中count++并不是原子操作,count++分成 读–>修改–>写 三步组成,所以在多线程的访问下,该操作不是线程安全的.因此我们必须采用同步机制来使count++为原子操作.

①让方法为原子操作: 在increment方法前加入synchronized.
public synchronized void increment(){
    count++;
}


②使用synchronized包含将要同步的代码块: 可以让count++这段代码为原子操作.
public void increment(){
    synchronized(this){
        count++;
    }
}

上面两种方法方法都能让线程同步访问,并且两者用的同步锁都是this对象.在线程同步(一)中介绍过当多个线程mutex使用的是同一对象时,那么多个线程间就达到了互斥的效果.在实例方法上使用synchronized关键字,那么使用的锁的对象就是this.若increment方法为静态方法,在静态方法上面使用synchronized关键字,这样锁对象就是类似与类的字节码.所以下面两段代码为等效的同步(只是同步等效,方法并不等效,即对于该类的所有对象这两种方法对于多个线程同一时刻只能一个线程进入该同步块,而上面代码块是对于一个对象,多个线程同一时刻只能一个线程进入该同步块).

public synchronized static void increment(){
    count++;
}

public void increment(){
    synchronized(Counter.class){
        count++;
    }
}

      由于使用不同的锁从而达到的互斥的效果也不相同,对于多个对象,多个线程间的互斥代码我们应该使用静态变量的锁,而对于一个对象,多个线程间的互斥应该使用实例锁.上面使用synchronized方法和synchronized代码块都使用的内置锁加锁机制.这种锁类型最重要的一个特点就是Reentrancy(可重入).什么是可重入呢?
      在了解可重入之前,想想当一个线程进入critical section JVM是通过什么机制让其他线程不能进入呢?其实是通过一个计数器和一个绑定的线程,当一个线程想进入critical section 时,先判断计数器是否等于0,若为0就让线程进入并绑定该线程计数器+1,这时,若其他线程要进入则会阻塞在外面.所以每一个锁都关联一个所有者的线程和一个计数器.

      当i=2时,首先获取锁对象进入方法内部这时该锁与该线程关联,并计数器值为1,当i=1时,这时在获取锁时,如果JVM判断是同一个锁对象并且是同一个线程那么线程就可再次进入的并计数器的值+1.当线程退出一个同步块计数器-1,当计数器为0时,该锁就自动释放了.上面体现了锁的可重入性.下面我们看看可重入的另外的一中情况.
