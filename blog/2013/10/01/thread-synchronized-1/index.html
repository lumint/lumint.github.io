线程同步(一)

Oct 1st, 2013

介绍

1.1线程是什么?要弄清楚线程是什么,首先需要知道另一个名词进程,那什么是进程呢?
       简单的来讲,进程就是一次程序(指的是源代码通过编译,汇编,连接等变成的可执行文件)的执行.它是一个动态的过程,而程序却是一个静态的概念. 比如:一个聊天软件QQ放在桌面你没有去点击时,这时就是一个存放在硬盘的程序,通过你的点击开始加载到内存开始执行到最终退出,这个中间过程就是程序的一次执行,就是所谓的进程.(还有文本编辑器,Web Browser等).居然进程是程序的一次执行,那么进程和程序有什么区别呢?

1.2进程和程序的区别?
       进程虽然来至于程序,但是它和程序有很大的不同.最主要的前者是动态的而后者却是静态的.进程不仅仅包含text section(程序代码区),还必须有指向运行时的程序计数器,还包含程序运行stack(存放一些函数参数,返回地址,局部变量等),还有数据区存放一些全局变量,还有Heap(堆区)存放程序运行过程中动态分配和创建的一些数据.如图:

线程

      明白进程的概念以后,然后在看看什么是线程,线程是操作系统的基本调度单位,它包含线程ID,程序计数器,一组register, a stack. 一个进程包含一个或多个线程.同一基础的多个线程间共享进程的code section(代码区),data section和其他一些操作系统资源(IO等), 如果一个进程中含有多个线程,那么每次它可以完成多个任务.如图:
              
                                                                    图2:单线程和多线程进程
      多线程的出现大大提高了CPU的利用率,然后由于多线程的编程,出现多个线程访问同一资源,多个线程写同一数据, 这样就带来了共享资源的数据不一致和错误,所以我们必须通过一些机制来实现线程间的同步和通信.

      为了同一时间只允许一个线程访问共享资源,提出了Critical Section(临界区)的概念.Critical Section 是同一时间最多只允许一个线程进入的代码块.
      在各种编程语言中几乎都提供了Critical Section的实现(由于笔者对Java比较熟,所以下面以Java为例),当一个线程要访问Critical Section时,根据同步机制来检查是否有其它的线程进入Critical Section.如果没有则该线程进入Critical Section,否则,这个线程必须等到当前进入Critical Section的线程从Critical Section退出,才能进去.如果有多个线程在等待,那么JVM就调度其中一个线程(根据一定的策略[优先级,响应比等])进入Critical Section.
      Java采用Synchronized关键字和Lock对应的API实现(Java1.5↑)线程的同步问题.下面我们就使用synchronized和Lock来实现线程的同步问题.

      当一个线程进入critical section时,那么JVM是根据什么来判断该线程是否能进入呢?当编写程序设定critical section时,我们会为该critical section指定一个mutex(互斥量),而互斥量的值将决定 其他线程是否可以进入该Critical Section. 

        synchronized(mutex){
            // critical section code
            ...
        }

      当程序运行开始由于线程一,线程二,线程三使用的同一个mutex,当线程一进入Critical Section,线程二和线程三将被阻塞在critical section外面,而线程四使使用的mutex2互斥量将不会阻塞,所以先是线程一和线程四进入Critical Section(因为线程一和线程四不是使用的同一mutex导致Critical Section能被多个线程进入). 当线程一运行完成,线程二和线程三将会同时竞争进入Critical Section,JVM根据一定的策略来调度其一进去(线程调度的不确定性).在这里理解mutex是非常重要的,因为只有学会如何设定mutex才能让线程互斥访问.接下来将对mutex的使用做出讨论.
